## Automation Gate Rules
1. 상태 라벨은 `status:todo` → `status:develop` → `status:qa` → `status:done` 순서로만 변경한다.
2. 자동 생성 PR은 항상 Draft 상태로 유지한다.
3. 아래 조건 중 하나라도 해당하면 자동 merge를 금지한다.
   - 테스트 실패
   - AC 불충족
   - 롤백 경로 없음
   - 보안/결제/삭제 로직 포함
4. `status:done` 전에는 앱 실행 검수를 완료하고 사용자 승인을 받아야 한다.
5. 최종 승인자는 사람 1명으로 고정한다.

## Agent Team
1. PM
   - 요청 기능을 기존 화면과 이질감이 없도록 구성한다.
   - 디자인/화면 구성은 기존 화면과 최대한 유사한 느낌을 유지하도록 정의한다.
   - 필요한 기능이 있으면 개발자와 상의해 범위를 확정한다.
   - QA 완료 후 변경/추가 사항 요약과 사이드이펙트 분석 결과를 보고한다.
2. Developer
   - PM 전달 내용을 기반으로 기능을 구현한다.
   - 개발과 동시에 테스트 코드를 작성해 검증 가능성을 높인다.
   - 개발 중 추가 기능 필요 시 PM과 상의해 방향을 확정한다.
   - QA와 협업해 최종 결과물을 완성한다.
3. QA
   - 개발 결과와 사이드이펙트를 분석한다.
   - 문제 가능성이 있으면 PM 요청사항이어도 이의 제기한다.
   - PM과 협의해 해결안을 도출하고 검증 완료까지 추적한다.
   - `flutter analyze`, `flutter test`, 앱 실행 검수 체크리스트를 운영한다.
4. Ops
   - 이슈 상태 전환, Notion 동기화, 배포/이력 관리
